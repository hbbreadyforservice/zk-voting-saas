# VoteCloud ZK Ceremony Guide

This guide explains how VoteCloud should move from the local development setup to a production-grade zk-SNARK ceremony.

## Why This Matters

The current local setup is useful for demos, but it is not enough for production. A Groth16 circuit needs:

1. A reusable Phase 1 Powers of Tau file.
2. A circuit-specific Phase 2 ceremony.
3. Public verification of the final `.zkey`.

VoteCloud uses a Merkle membership + nullifier voting circuit. If the final proving key is generated by one trusted party, that party could theoretically keep toxic waste. A multi-party ceremony reduces this risk because the setup remains secure if at least one participant destroys their contribution secret.

## Files

Important files:

```txt
circuits/voting.circom
circuits/setup.sh
circuits/setup-production.sh
circuits/verify-ceremony.sh
backend/src/zkfiles/
frontend/public/zkfiles/
contracts/Verifier.sol
```

## Development Setup

For local demos only:

```bash
cd circuits
./setup.sh
```

This generates a local Powers of Tau ceremony. Do not use it in production.

## Production-Oriented Setup With Hermez Phase 1

Use:

```bash
cd circuits
./setup-production.sh
```

This script downloads:

```txt
https://storage.googleapis.com/zkevm/ptau/powersOfTau28_hez_final_15.ptau
```

The expected BLAKE2b hash is:

```txt
982372c867d229c236091f767e703253249a9b432c1710b4f326306bfa2428a17b06240359606cfe4d580b10a5a1f63fbed499527069c18ae17060472969ae6e
```

The script:

1. downloads the `.ptau` file if missing;
2. verifies its BLAKE2b hash;
3. compiles `voting.circom`;
4. runs Groth16 setup;
5. contributes local entropy;
6. applies a final beacon;
7. exports `verification_key.json`;
8. exports `contracts/Verifier.sol`;
9. copies artifacts to backend and frontend;
10. writes `circuits/build/ceremony-metadata.json`.

This is stronger than a local Phase 1, but a real production launch should still run a public multi-party Phase 2.

## True Production Phase 2 With p0tion

p0tion is a ceremony coordination tool from Privacy & Scaling Explorations. Use it to coordinate multiple contributors for the circuit-specific Phase 2.

### 1. Prepare The Circuit

Compile the circuit first:

```bash
cd circuits
circom voting.circom --r1cs --wasm --sym --output build -l ../node_modules
npx snarkjs r1cs info build/voting.r1cs
```

Record:

```txt
number of constraints
number of public inputs
SHA256 of voting.r1cs
SHA256 of voting.circom
git commit hash
```

### 2. Choose Phase 1 File

Use Hermez:

```bash
mkdir -p circuits/ptau
curl -L https://storage.googleapis.com/zkevm/ptau/powersOfTau28_hez_final_15.ptau \
  -o circuits/ptau/powersOfTau28_hez_final_15.ptau
```

Verify:

```bash
node -e "const fs=require('fs');const crypto=require('crypto');console.log(crypto.createHash('blake2b512').update(fs.readFileSync('circuits/ptau/powersOfTau28_hez_final_15.ptau')).digest('hex'))"
```

Expected:

```txt
982372c867d229c236091f767e703253249a9b432c1710b4f326306bfa2428a17b06240359606cfe4d580b10a5a1f63fbed499527069c18ae17060472969ae6e
```

### 3. Create Initial zkey

```bash
cd circuits
npx snarkjs groth16 setup \
  build/voting.r1cs \
  ptau/powersOfTau28_hez_final_15.ptau \
  build/voting_0000.zkey
```

### 4. Coordinate Contributors

Recommended participant profile:

- at least 5 independent participants;
- different machines;
- at least one participant not affiliated with the developer;
- each participant records their contribution hash;
- participants delete local entropy after contribution.

For a university project, acceptable participants could include:

- supervisor;
- classmates;
- one external developer;
- project owner;
- one independent reviewer.

### 5. Manual snarkjs Contribution Flow

If not using p0tion, participants can contribute sequentially:

Participant 1:

```bash
npx snarkjs zkey contribute \
  build/voting_0000.zkey \
  build/voting_0001.zkey \
  --name="participant-1" \
  -v
```

Participant 2:

```bash
npx snarkjs zkey contribute \
  build/voting_0001.zkey \
  build/voting_0002.zkey \
  --name="participant-2" \
  -v
```

Continue until the final participant.

### 6. Apply Final Beacon

Use a public random beacon. Examples:

- future Ethereum block hash;
- NIST randomness beacon;
- public lottery/randomness source chosen before it is known.

Example:

```bash
npx snarkjs zkey beacon \
  build/voting_000N.zkey \
  build/voting_final.zkey \
  <PUBLIC_RANDOM_HEX_STRING> \
  10 \
  -n="VoteCloud final beacon"
```

### 7. Export Verification Files

```bash
npx snarkjs zkey export verificationkey \
  build/voting_final.zkey \
  build/verification_key.json

npx snarkjs zkey export solidityverifier \
  build/voting_final.zkey \
  ../contracts/Verifier.sol
```

### 8. Verify Final zkey

```bash
cd circuits
./verify-ceremony.sh
```

Also run:

```bash
npx snarkjs zkey verify \
  build/voting.r1cs \
  ptau/powersOfTau28_hez_final_15.ptau \
  build/voting_final.zkey
```

### 9. Copy Artifacts

```bash
mkdir -p ../backend/src/zkfiles ../frontend/public/zkfiles
cp build/voting_js/voting.wasm ../backend/src/zkfiles/
cp build/voting_final.zkey ../backend/src/zkfiles/
cp build/verification_key.json ../backend/src/zkfiles/
cp build/voting_js/voting.wasm ../frontend/public/zkfiles/
cp build/voting_final.zkey ../frontend/public/zkfiles/
cp build/verification_key.json ../frontend/public/zkfiles/
```

## Production Acceptance Checklist

Before production, verify:

- `circuits/voting.circom` is final.
- Circuit parameters are final, especially Merkle tree depth.
- `voting.r1cs` hash is recorded.
- Hermez `.ptau` hash is verified.
- At least 5 independent Phase 2 contributions are recorded.
- Final beacon source is public and chosen before it is known.
- `snarkjs zkey verify` passes.
- `Verifier.sol` is regenerated from the final zkey.
- Contracts are redeployed with the final verifier.
- Backend/frontend artifacts are copied from the same final zkey.
- `LOCALMode` is disabled.
- `/api/voter/generate-proof` is removed or disabled.
